## RESUMEN EJECUTIVO

El producto tiene una arquitectura técnica más sólida que el promedio de MVPs en su etapa, buena separación de capas y un backend con autenticación correctamente implementada. Sin embargo, hay **cinco problemas que pueden hundir el negocio antes de que despegue**, y son todos solucionables:

1. **Identidad de producto esquizofrénica** — hay 4 nombres distintos para el mismo producto y 2 propuestas de valor incompatibles en el mismo landing.
2. **Los precios del landing no existen en el código** — un usuario que intente comprar el "Plan PYME" ($14.990) no recibirá nada porque ese precio no está mapeado.
3. **El webhook de MercadoPago no tiene firma HMAC** — cualquiera puede activar planes gratis desde fuera.
4. **Las credenciales de SII no están implementadas** — el feature core del landing (F29 automático) no tiene dónde guardar las claves del usuario.
5. **Se prometen features que no existen** — WhatsApp, monitoreo diario, F29 automático. Ninguno está operativo.

## 1. 🔴 SEGURIDAD — HALLAZGOS CRÍTICOS

### 1.1 🔴 CRÍTICO — Webhook MercadoPago sin verificación de firma HMAC

**Archivo:** `frontend/src/app/api/webhooks/mercadopago/route.ts`

El endpoint recibe cualquier POST, consulta el pago en la API de MP y actualiza el plan. No verifica la firma `x-signature` que MercadoPago envía en cada webhook.

**Vector de ataque:**
```bash
# Atacante hace un pago real de $1 en MP (otro producto), obtiene un payment_id real
# Envía ese ID al webhook de Tramita con metadata.user_id de cualquier víctima
curl -X POST https://dejadwebiar.vercel.app/api/webhooks/mercadopago \
  -H "Content-Type: application/json" \
  -d '{"type":"payment","data":{"id":"PAYMENT_ID_REAL_DE_MP"}}'
```

Si `payment.status === 'approved'` y el monto coincide, el plan se activa sin que nadie haya pagado a Tramita.

**Fix requerido:**
```typescript
// Al inicio del handler, verificar:
const signature = request.headers.get('x-signature');
const requestId = request.headers.get('x-request-id');
// Calcular HMAC-SHA256 con MP_WEBHOOK_SECRET y comparar
```

### 1.4 🟠 ALTO — Ausencia de Content-Security-Policy y HSTS

**Archivo:** `frontend/vercel.json`

El vercel.json tiene X-Frame-Options y X-Content-Type-Options pero le faltan:
- `Content-Security-Policy` — protección principal contra XSS
- `Strict-Transport-Security` — fuerza HTTPS en clientes
- `X-Permitted-Cross-Domain-Policies`

Un producto que maneja credenciales tributarias y datos financieros **necesita CSP obligatoriamente**.

### 1.5 🟠 ALTO — Logs exponen datos sensibles en Render

**Archivo:** `backend_node/src/server.js`

```javascript
logger.info(`${req.method} ${req.path}`, { ip: req.ip, body: req.body });
```

El body completo de cada request va a los logs. Si el request incluye un RUT o credenciales, quedan persistidos en los logs de Render en texto plano. En producción, el body nunca debe loguearse directamente — solo campos no sensibles.

### 1.7 🟡 MEDIO — CORS backend permite origen vacío (server-to-server sin restricción)


Esta línea está bien en desarrollo pero en producción permite que cualquier cliente que no envíe un header `Origin` (curl, scripts, Postman) acceda sin restricción. Si `SCRAPER_API_KEY` se filtra, combinado con esto, hay acceso total.

**Archivo:** `frontend/src/services/copilot.ts`

## 2. 🔴 MODELO DE NEGOCIO — INCONSISTENCIAS FATALES

### 2.1 🔴 CRÍTICO — Precios del landing NO coinciden con el código

**El landing muestra:**
- Plan Ciudadano: $0
- Plan PYME: **$14.990/mes**
- Plan Contadores: **$39.990/mes**

**El código implementa:**
```typescript
// types.ts
export const PLAN_PRICES = {
  basic:   { clp: 3990,  label: '$3.990/mes' },
  premium: { clp: 7990,  label: '$7.990/mes' },
}

// webhooks/mercadopago/route.ts
const PRICE_TO_PLAN: Record<number, Plan> = {
  3990: 'basic',
  7990: 'premium',
}
```

**Resultado:** Si un usuario hace clic en "Obtener Plan PYME" ($14.990), el sistema intentará crear una preferencia de MP sin plan mapeado, o el webhook recibirá $14.990 y no encontrará ningún plan correspondiente — el usuario habrá pagado y no recibirá nada. **Esto es un bug de fraude involuntario.**

---

### 2.2 🔴 CRÍTICO — El feature principal del landing no está implementado

**Lo que el landing promete:**
- "Tu F29 mensual listo a tiempo, incluso si se te olvida" (H1)
- "Preparamos y presentamos de forma automática tus borradores de Formulario 29"
- "Supervisamos diariamente tus obligaciones automáticamente en background"
- "Recibes comprobantes del SII directamente a tu WhatsApp"
- "Notificaciones inmediatas en WhatsApp"

**Lo que realmente existe:**
- ✅ Búsqueda manual de causas PJUD
- ✅ Consulta manual SII/TGR (si el usuario lo pide)
- ✅ Chat IA para "inicio de actividades SII" (flujo limitado)
- ❌ F29 automático (requiere credenciales del SII — no hay dónde guardarlas)
- ❌ Monitoreo diario en background (no hay cron job implementado)
- ❌ WhatsApp (no hay integración)
- ❌ Declaración automática (requeriría Clave Tributaria del SII)

**El landing vende un producto que no existe.**

### 2.3 🟠 ALTO — No hay infraestructura para almacenar credenciales tributarias

La propuesta de valor de Tramita requiere que el sistema acceda al SII en nombre del usuario. Para eso necesita la **Clave Tributaria del SII** del usuario. 

El schema de Supabase tiene `profiles` con nombre, RUT, teléfono — pero **no hay tabla ni campo para credenciales del SII**. El tipo `UserCredentials` en types.ts existe pero no hay migración correspondiente.

El landing dice "AES-256-GCM" para la encriptación, hay un `CREDENTIALS_ENCRYPTION_KEY` en el .env, pero no hay código que use esa clave para cifrar/descifrar credenciales del usuario. La promesa de seguridad es marketing sin implementación.

El documento `TRAMITAI_Estrategia_Negocio.md` describe un producto de búsqueda judicial B2C. El landing actual vende automatización tributaria. La estrategia habla de $3.990 y $7.990 — el landing cobra $14.990 y $39.990. Son mundos distintos sin un documento de reconciliación.

## 3. 🟠 WORKFLOW — PROBLEMAS TÉCNICOS Y DE FLUJO

### 3.1 🟠 ALTO — El mensaje del usuario no se persiste antes del procesamiento

**Archivo:** `frontend/src/services/copilot.ts`

El flujo del chat:
1. Usuario envía mensaje → API route
2. Se llama `procesarTurnoCognitivoSoberano`
3. Se llama a Claude (puede tardar hasta 60s)
4. **Solo al final se guarda la respuesta del assistant**

El mensaje del usuario NUNCA se guarda. Si la request falla a mitad (timeout, error de red, crash), el usuario pierde el contexto. No hay persistencia de mensajes de usuario en `chat_messages`.

---

### 3.2 🟠 ALTO — `consultar_tgr` está declarado en el permission matrix pero no implementado

```typescript
// TOOL_ACCESS_BY_WORKFLOW
'cobranza_tgr': ['consultar_tgr', 'update_workflow_state'],
```

La herramienta `consultar_tgr` está autorizada para el flujo `cobranza_tgr` pero no hay ninguna función `consultarTGR` implementada en copilot.ts. Si la IA intenta usarla, el sistema fallará silenciosamente o lanzará un error.

---

### 3.3 🟡 MEDIO — El middleware tiene un matcher demasiado amplio

**Archivo:** `frontend/src/middleware.ts`

```typescript
matcher: [
  '/((?!_next/static|_next/image|favicon.ico|.*\\.(?:png|svg|jpg|ico|webp)).*)',
]
```

El middleware se ejecuta en prácticamente TODAS las rutas, incluyendo las API routes (`/api/*`). Esto es ineficiente y puede causar edge cases. Las API routes tienen su propia autenticación — el middleware debería limitarse solo a rutas UI.

---

### 3.4 🟡 MEDIO — La función de quota no actualiza el contador correctamente

**Archivo:** `frontend/src/app/api/buscar/route.ts`

```typescript
// Se verifica la cuota ANTES de buscar
const { data: monthlyCount } = await supabase.rpc('get_monthly_search_count', { p_user_id: user.id })

// Se guarda la búsqueda DESPUÉS de buscar (paso 7)
// Se recuenta DESPUÉS de guardar (paso 8)
```

Hay una race condition si el usuario hace dos requests simultáneas: ambos pasan la verificación de cuota (ambos ven el mismo contador), ambos ejecutan el scraper, y luego ambos incrementan el contador. Un usuario free con 2 consultas restantes puede ejecutar múltiples búsquedas en paralelo.

## 4. 🟠 EXPERIENCIA DE USUARIO — PROBLEMAS GRAVES

### 4.1 🔴 CRÍTICO — 4 nombres distintos para el mismo producto

A lo largo del código, documentación y UI se usan indistintamente:
- **"Tramita"** — el nombre en el landing, navbar, footer, CTAs, mensajes de WhatsApp mock
- **"DJADWEB-IA®"** — nombre de marca registrada según la carpeta del proyecto
- **"DejadWebiar"** / **"DEJADWEBIAR"** — nombre del repositorio y dominio
- **"DW Trámites y Tecnología SpA"** — razón social del footer

Un usuario que llega al producto no sabe cómo se llama. El copilot se presenta como "DJADWEB-IA®" pero el producto que lo contiene dice "Tramita". Esto destruye la confianza y el recall de marca.

**Decisión urgente:** elegir UN nombre y ser consistente en absolutamente todo.

---

### 4.2 🟠 ALTO — El landing vende F29/SII pero el producto core sigue siendo búsqueda PJUD

El H1 del landing habla de "tu F29 mensual". Los 3 feature cards hablan de declaración de IVA, supervisión de deuda fiscal y monitoreo judicial ROL. Pero si el usuario se registra y entra al sistema, el producto principal es **búsqueda de causas judiciales por nombre** — que no tiene nada que ver con F29.

El producto tiene un **pivot a medias**: el messaging cambió hacia tributario pero la funcionalidad sigue siendo judicial. Esto crea una experiencia de usuario totalmente frustrante — el usuario que viene buscando "F29 automático" llega a un buscador de causas judiciales.

---

### 4.3 🟠 ALTO — El mockup del dashboard usa datos que parecen reales

El hero del landing muestra un mockup de dashboard con:
- "Inversiones Soto & Ruiz Ltda. • RUT: 76.982.104-K"
- "F29 IVA Abril 2026 — Presentado el 12 de Mayo de 2026 — FOLIO 928310"
- "F29 IVA Mayo 2026 — Vence el 12 de Junio de 2026 — BORRADOR LISTO"
- Un chat de WhatsApp que dice "Andrea, tu declaración F29 del periodo 05-2026 fue presentada"

Para un usuario no técnico, esto parece una captura de pantalla real de una empresa cliente. En realidad es HTML hardcodeado. Si alguien lo interpreta como evidencia de funcionamiento y luego el producto no puede hacer lo que muestra, es **publicidad engañosa**.

---

### 4.4 🟡 MEDIO — No hay onboarding tras el registro

Un usuario se registra → llega al dashboard → ¿qué hace? No hay:
- Tour del producto
- Pasos guiados para configurar el RUT
- Explicación de qué puede hacer con su plan free
- Llamada a la acción clara

El gap entre "registrarse" y "obtener valor" es completamente invisible.

---

### 4.5 🟡 MEDIO — Markdown literal en el landing

**Archivo:** `frontend/src/app/page.tsx`, línea 224

```typescript

```

Los asteriscos de Markdown se renderizan en texto plano porque es JSX, no un parser Markdown. El usuario ve `**AES-256-GCM**` literal con asteriscos. Es un bug de presentación visible para todos.

## 5. ANÁLISIS ARQUITECTURAL — LO QUE SÍ ESTÁ BIEN

Para ser justos, hay partes bien construidas:

✅ **Separación frontend/backend** — La arquitectura de proxy (Next.js → backend Node.js en Render) es correcta y protege el scraper.

✅ **Row Level Security en Supabase** — Las políticas RLS están bien definidas. Cada usuario solo accede a sus datos.

✅ **Autenticación correcta en la mayoría de las rutas** — `/api/buscar` y `/api/chat` verifican `supabase.auth.getUser()` correctamente.

✅ **Rate limiting en backend** — 10 req/min por IP es razonable para un scraper con Playwright.

✅ **Agentic loop con límite de iteraciones** — MAX_TOOL_ITERATIONS = 6 previene bucles infinitos en el copilot.

✅ **Tool Policy Layer en copilot** — La validación de herramientas por workflow es sofisticada y bien pensada.

✅ **Correlation IDs en copilot** — Trazabilidad vertical implementada correctamente.

✅ **Caché LRU con TTL en scraper PJUD** — pjud.js tiene un sistema de caché inteligente.

✅ **Triggers Supabase para onboarding** — El trigger `on_auth_user_created` crea perfil + suscripción free automáticamente.

## 7. VEREDICTO FINAL

**El producto tiene huesos sólidos pero está vendiendo un sueño que no puede entregar todavía.**

La arquitectura técnica es más madura que el promedio de proyectos en esta etapa. Sin embargo, el gap entre lo que el landing promete y lo que el sistema puede hacer es demasiado grande para ignorarlo. Un usuario que pague el Plan PYME ($14.990/mes) esperando F29 automático y notificaciones WhatsApp va a pedir reembolso en 48 horas.

**La decisión más importante que hay que tomar NO es técnica — es de producto:** ¿Tramita es una herramienta de búsqueda judicial o un automatizador de impuestos? Estas son propuestas de valor completamente distintas con usuarios distintos, workflows distintos y regulaciones distintas. Intentar ser los dos al mismo tiempo con los recursos actuales garantiza hacer ambas cosas a medias.

**Recomendación:** Congelar el landing en lo que el producto SÍ hace hoy (búsqueda judicial + alertas), cobrar $3.990/$7.990 como dice el código, y construir el F29 automático como el siguiente hito mensurable. Lanzar con honestidad es más sostenible que prometer y fallar.
